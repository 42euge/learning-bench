Trace-Based Imitation

Observational Learning

What It Tests

Can the model learn a novel procedure by observing worked examples (execution traces), then apply it to a new input? This tests observational learning—inferring a procedure from demonstrations without explicit instruction.

Three-Phase Framework

📝
Pre-Test
See worked examples of a procedure on number lists, apply to a new input cold.
→
📖
Study
Analyze traces to write down the exact procedure, order of operations, conditions.
→
✅
Post-Test
Apply procedure with own notes + original traces.

Difficulty Dimensions

DimensionLevelsWhat Varies
Complexity simple, medium, complex Simple: 1-step procedures (zigzag_sum, mirror_diff, triangle_count). Medium: 2-step compositions (spiral_crunch, wave_hash, cascade_mod). Complex: 3+ steps with conditional branching (branch_fold, pivot_weave, chain_gate).
Traces 2, 3, 4 Number of worked examples provided. More traces = easier to identify the procedure.
Seeds 0, 1, 2 Independent random generations.

Total: 3 complexity × 3 traces × 3 seeds = 27 items

Dataset Generation

Nine novel procedures are defined with increasing complexity. Each transforms a list of integers into a single integer through specific operations. Worked traces are generated by running each procedure on random inputs and formatting each step explicitly. The test input is always different from any traced input.

Example procedures:

  • zigzag_sum — Alternate adding and subtracting elements left-to-right
  • wave_hash — Sort, take differences of consecutive elements, then sum
  • branch_fold — If first element is even, fold left with addition; else fold right with subtraction

Scoring

  • Metric: Post-learning accuracy (exact integer match).
  • Extraction: Last integer found in the response (handles negatives).
  • Assertion: kbench.assertions.assert_equal(post_extracted, str(expected))

What Discriminates Models

  • Simple/4 traces: Most models identify single-step operations from 4 examples.
  • Complex/2 traces: Conditional branching with only 2 traces requires genuine procedural reasoning—figuring out what triggers different code paths.
  • Precision matters: Procedures like cascade_mod involve modular arithmetic chains where off-by-one errors cascade.

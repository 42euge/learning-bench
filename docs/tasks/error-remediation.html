Error Remediation

Associative Learning

What It Tests

Can the model learn the remediation procedures of an invented CLI tool from a few worked examples, instead of hallucinating plausible-looking flags? We introduce zkl, a fictional package/deploy CLI whose errors follow a structured grammar (error-code prefix = category; keywords = specific fix). The model studies a short troubleshooting manual, then must emit the correct remediation command for a new error from the same tool.

Three-Phase Framework

📝
Pre-Test
See a novel error message and guess the remediation cold (no manual).
→
📖
Study
Read a troubleshooting manual excerpt; group errors by family, extract patterns, build a lookup table.
→
✅
Post-Test
Answer with own study notes + original manual. Must produce the exact zkl command.

Difficulty Dimensions

DimensionLevelsWhat Varies
Complexity direct_lookup, pattern_match, multi_factor Direct: exact error code → fixed remediation (1:1 lookup). Pattern: code family prefix + keyword in message → verb + target. Multi-factor: code family AND context qualifier (cold-start vs hot-path) jointly select remediation verb, target, and flag.
Evidence few (5), mid (7), many (10) Number of error–remediation pairs in the manual excerpt. More pairs provide more coverage but also more to process.
Seeds 0, 1, 2 Independent random generations for statistical robustness.

Total: 3 complexity × 3 evidence × 3 seeds = 27 items

Dataset Generation

The fictional tool zkl has three error families: E1xxx (network/registry), E2xxx (filesystem/cache), and E3xxx (auth/credentials). Each family maps to a specific remediation verb. For direct_lookup, exact error codes map 1:1 to exact commands (e.g., E1001 → zkl registry refresh --scope global). For pattern_match, the family prefix plus a keyword in the message body determines the remediation noun. For multi_factor, both the family AND a context qualifier (cold-start vs hot-path) must be considered jointly to select the correct verb, target, and flag. Test errors use new message templates and targets so recognition must come from the learned patterns, not surface matching.

Examples

Direct Lookup (1:1 code mapping)

The model sees a troubleshooting manual with exact error–remediation pairs, then must recall one:

ERROR: E1001: operation failed during package sync.
FIX:   zkl registry refresh --scope global

ERROR: E2003: operation failed during workspace open.
FIX:   zkl cache purge --deep

ERROR: E3002: operation failed during session bootstrap.
FIX:   zkl auth rotate --keyring
New error: [E1001] dependency fetch did not complete.
Expected: zkl registry refresh --scope global

The error code E1001 appeared in the manual with its exact fix. The test uses a different message template and context, so the model must recognize the code, not the sentence.

Pattern Match (family + keyword)

Remediation depends on the error family prefix and a keyword in the message:

ERROR: E1204: mirror check failed for target “acme-core”.
FIX:   zkl refresh mirror

ERROR: E2317: inode subsystem reports invalid state at “bluefin-utils”.
FIX:   zkl purge index

ERROR: E3158: keyring reconciler on “slate-auth”.
FIX:   zkl rotate keyring
New error: E1440: mirror subsystem reports invalid state at “glacier-net”.
Expected: zkl refresh mirror

The specific code E1440 is new, but the E1 family (“refresh”) and keyword “mirror” were both shown—the model must apply the pattern rule.

Multi-Factor (family + qualifier)

Both the error family and a context qualifier jointly determine the remediation:

ERROR: E1305: failure on cold-start pathway for “north-star”.
FIX:   zkl bootstrap registry --seed

ERROR: E1422: failure on hot-path pathway for “pivot-tools”.
FIX:   zkl reconnect registry --keepalive

ERROR: E2187: cold-start run aborted while touching “ember-cli”.
FIX:   zkl rebuild cache --from-manifest
New error: E2533: hot-path executor handling “quartz-runtime”.
Expected: zkl compact cache --online

Using only the family (E2) would suggest “rebuild cache”, and using only the qualifier (“hot-path”) is ambiguous. The model must use both factors jointly to produce the correct remediation with the right flag.

Scoring

  • Metric: Post-learning accuracy (exact command match).
  • Extraction: Last line starting with “zkl”, lowercased and stripped of formatting.
  • Assertion: kbench.assertions.assert_equal(post_extracted, expected)

What Discriminates Models

  • Hallucinated flags: Models that ignore the study notes and guess from training priors will produce plausible-looking but wrong commands (e.g., --libgs, --no-turbopack). Only models that actually learn the mapping succeed.
  • Direct vs pattern: Direct lookup is rote recall; pattern matching requires abstracting the family–verb and keyword–noun rules from examples.
  • Multi-factor: The hardest condition—models that latch onto a single factor (family OR qualifier) will be wrong on disambiguating test cases where both matter.
  • Study quality matters: Models that produce organized lookup tables outperform those that generate vague summaries of the manual.

Prior Work & References

Error remediation draws on fault localization in software engineering and the growing body of work on LLM-based debugging. No existing benchmark tests whether models can learn an invented tool’s error–fix mappings from a short manual—most evaluate against real codebases where prior training knowledge is a confound.

Cognitive Science Foundations

  • Stimulus–Response Learning (Thorndike, 1911). The law of effect: responses followed by satisfaction are strengthened. Error remediation is fundamentally an S–R mapping task: specific error patterns (stimuli) must be associated with specific fix commands (responses).
  • Paired-Associate Paradigm (Calkins, 1894). Our error–fix pairs are a domain-specific instantiation of the classic paired-associate method, where arbitrary stimuli are paired with arbitrary responses and the learner must recall the correct response given the stimulus.

Fault Localization & Program Repair

  • Jimenez et al. (2024). “SWE-bench: Can Language Models Resolve Real-world GitHub Issues?” (ICLR 2024). The leading benchmark for LLM-based software repair. Unlike SWE-bench, which tests against real repositories where models may have memorized patterns, our task uses a wholly invented tool to isolate in-context learning from prior knowledge.
  • Yang et al. (2026). “On the Role of Fault Localization Context for LLM-Based Program Repair”. Showed that file-level localization context yields a 15–17x improvement over a no-file baseline. Our task similarly tests whether models can use provided context (the manual) to localize the correct fix.
  • Xie et al. (2026). “RGFL: Reasoning Guided Fault Localization for Automated Program Repair”. Introduced hierarchical reasoning for fault localization. Our multi-factor condition similarly requires hierarchical reasoning (family + qualifier) to select the correct remediation.

LLM Tool Use & Error Diagnosis

  • Ye et al. (2024). “DebugBench: Evaluating Debugging Capability of Large Language Models”. A benchmark of 4,253 bugs across C++, Java, and Python. Tests bug fixing in real languages; our task tests whether the model can learn a new tool’s conventions from examples rather than relying on prior knowledge of real tools.
  • Huang et al. (2024). “SpecTool: A Benchmark for Characterizing Errors in Tool-Use LLMs”. Identified seven common error patterns in LLM tool-use tasks. Our benchmark specifically targets one of these: hallucinated tool arguments (flags and options the model invents rather than learning from documentation).
  • Guo et al. (2025). “Where LLM Agents Fail and How They Can Learn From Failures”. Studied failure recovery in agentic systems. Our task isolates the core capability: can the model learn the correct recovery action from a manual, rather than persevering with incorrect guesses?

Log Analysis & Automated Remediation

  • LogSage (2025). “LogSage: An LLM-Based Framework for CI/CD Failure Detection and Remediation”. An end-to-end framework using RAG for root cause analysis. Our task strips away the retrieval component to test whether the model can learn the mapping directly from a short manual provided in-context.

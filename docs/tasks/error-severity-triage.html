Error Severity Triage

Concept Formation

What It Tests

Can the model learn a new tool’s severity classification system from a handful of labeled log lines and apply it to a novel line? The twist: models have a strong prior that anything labeled “error” is blocking. This benchmark tests whether a model can override that prior and correctly distinguish RETRYABLE_ERROR from TERMINAL_ERROR using only the evidence in the labeled examples.

Three-Phase Framework

📝
Pre-Test
See labeled log lines and classify a new line cold.
→
📖
Study
Build a decision procedure: identify surface features for each class, verify against every example.
→
✅
Post-Test
Classify with own study notes + original labeled examples.

Difficulty Dimensions

Tag-plus-code: a generic error tag plus a numeric code; the code range or prefix distinguishes retryable from terminal. Contextual: all lines share the same tag; severity depends on keywords in the message body.

Total: 3 complexity × 3 evidence × 3 seeds = 27 items

Dataset Generation

Log lines come from invented CLI tools (prismd, orbix, quarkctl, zephyrq, vantaloop, helixsync, nimboforge, tessera, lumenrun, glintd) with tool-specific severity conventions. For single_tag, each tool uses unique prefix tokens per class (e.g., @ok = INFO, @dead = TERMINAL_ERROR). For tag_plus_code, a generic error tag is shared but the attached code determines severity—e.g., codes 1xx–3xx are retryable while 4xx–9xx are terminal. For contextual, all lines share the same tag (STATUS: or >>) and severity depends entirely on message body keywords: “timeout” / “throttled” / “transient” signal retryable, while “corrupt” / “disk full” / “unauthorized” signal terminal. Test classes are stratified to ensure balanced probing of all four severities, with extra emphasis on the retryable vs. terminal distinction.

Examples

Single Tag (1:1 prefix mapping)

Each line has a unique prefix token that determines its severity class:

[.] fetch shard   → INFO
[~] resolve manifest   → WARNING
[*] compile pipeline   → RETRYABLE_ERROR
[!] flush segment   → TERMINAL_ERROR
[.] index ledger   → INFO
Classify: [*] validate cache-ring
Expected: RETRYABLE_ERROR

The [*] prefix was shown mapping to RETRYABLE_ERROR. The model must learn the prefix–class mapping and ignore the action/target content.

Tag Plus Code (error code determines retryable vs terminal)

A generic error tag plus a code; the code range distinguishes retryable from terminal:

OK :: fetch shard   → INFO
WARN :: resolve manifest   → WARNING
ERR :: compile pipeline [E-204]   → RETRYABLE_ERROR
ERR :: flush segment [E-501]   → TERMINAL_ERROR
ERR :: index ledger [E-133]   → RETRYABLE_ERROR
Classify: ERR :: validate cache-ring [E-455]
Expected: TERMINAL_ERROR

All three ERR lines share the same tag. The model must notice that lower codes (1xx–3xx) are retryable while higher codes (4xx–9xx) are terminal. Code E-455 is in the 4xx range, so it is TERMINAL_ERROR.

Contextual (body keywords determine severity)

All lines share the same tag; severity depends on message content:

STATUS: started fetch on shard   → INFO
STATUS: slow resolve on manifest (240ms)   → WARNING
STATUS: connection timeout reaching pipeline   → RETRYABLE_ERROR
STATUS: disk full: no space left while writing segment   → TERMINAL_ERROR
STATUS: throttled by ledger (try again)   → RETRYABLE_ERROR
Classify: STATUS: cache-ring manifest is corrupt
Expected: TERMINAL_ERROR

“Corrupt” signals a terminal condition in this tool’s conventions. A model relying on its prior that any error-ish word means “error” would miss the retryable vs. terminal distinction.

Scoring

  • Metric: Post-learning accuracy (exact severity label match).
  • Extraction: Scan response bottom-up for an exact token match (RETRYABLE_ERROR, TERMINAL_ERROR, WARNING, INFO), checking compound labels first to avoid partial matches.
  • Assertion: kbench.assertions.assert_equal(post_extracted, expected)

What Discriminates Models

  • Prior override: The core challenge. Models with a strong prior that “error” means terminal will misclassify retryable errors. Only models that learn from the examples can distinguish the two error classes.
  • Single-tag is easy: 1:1 prefix–class mappings are simple pattern matching. Most models get these right even pre-learning.
  • Tag-plus-code requires numeric reasoning: The model must infer a threshold rule (code range) from a few examples and apply it to an unseen code.
  • Contextual is hardest: Without any structural prefix to anchor on, the model must learn which specific keywords (timeout, throttled, corrupt, disk full) map to which severity class—and resist applying generic priors.
  • Study quality: Models that produce explicit decision procedures (“if keyword X, then class Y”) and verify them against every example outperform those that produce vague summaries.

Prior Work & References

Error severity triage adapts the concept formation paradigm to a practical domain: learning a new tool’s severity conventions from examples. The key insight is that LLMs carry strong priors about what “error” means, and the benchmark tests whether they can override those priors with in-context evidence.

Cognitive Science Foundations

  • Bruner, Goodnow & Austin (1956). A Study of Thinking. Concept attainment through classification: learners group instances into categories by discovering the relevant features. Our task is a four-class concept attainment problem where the features (tags, codes, keywords) define category boundaries.
  • Shepard, Hovland & Jenkins (1961). “Learning and memorization of classifications.” Showed that classification difficulty depends on rule complexity. Our three complexity levels (single_tag → tag_plus_code → contextual) mirror their type hierarchy: single-dimension rules are easiest, multi-cue rules are hardest.

Log Analysis & Error Classification

  • Chen et al. (2026). “LLM-Enhanced Log Anomaly Detection: A Comprehensive Benchmark”. Compared LLM-based and traditional approaches for log anomaly detection. Our task differs by testing whether models can learn a new tool’s conventions in-context, rather than applying knowledge of known log formats.
  • LogSage (2025). “LogSage: An LLM-Based Framework for CI/CD Failure Detection and Remediation”. Uses RAG for root cause analysis in CI/CD pipelines. Our task strips away retrieval to isolate the in-context category learning capability.
  • L4 (2025). “L4: Diagnosing Large-scale LLM Training Failures via Automated Log Analysis”. Analyzed 428 LLM training failures using taxonomy-based classification. Our task tests whether models can learn such a taxonomy from a handful of examples rather than having it pre-specified.

Automated Incident Response

  • IRCopilot (2025). “IRCopilot: Automated Incident Response with Large Language Models”. Automated incident detection and classification using LLMs. Our benchmark isolates the classification component: can the model learn severity conventions from examples without relying on prior knowledge of real incident-response systems?
  • Guo et al. (2025). “LLM-based Event Log Analysis Techniques: A Survey”. Comprehensive survey of LLM-based log analysis. Notes that LLMs often apply generic priors about severity rather than learning tool-specific conventions —exactly the failure mode our benchmark targets.

In-Context Learning & Prior Override

  • Min et al. (2022). “Rethinking the Role of Demonstrations” (EMNLP 2022). Found that ICL often relies on label space priors rather than learning from demonstrations. Our task directly tests whether models can override their “error = blocking” prior using in-context evidence.
  • Semantic Anchors (2025). “Why Small LLMs Cannot Flip Their Labels”. Showed that smaller models have near-zero semantic override rate—ICL operates as prior refinement. Our RETRYABLE_ERROR vs. TERMINAL_ERROR distinction specifically probes this: can the model override its default notion of what “error” means?
  • Wei et al. (2023). “Symbol Tuning Improves In-Context Learning” (EMNLP 2023). Our single_tag condition (novel tokens like [*] and [!]) is essentially a symbol-mapping task; symbol-tuned models would have an advantage.

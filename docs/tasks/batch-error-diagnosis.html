Batch Error Diagnosis

Observational Learning

What It Tests

Can the model identify ALL distinct root causes in a multi-error build output in one pass—not just the first one? This isolates observational learning: the ability to internalize a diagnostic procedure from worked examples and transfer it to a new multi-error scenario. The real-world failure mode is agents that fix one error, re-run, see the next, fix it, re-run… doing N build cycles instead of 1.

Three-Phase Framework

📝
Pre-Test
See worked examples of multi-error diagnosis, then count root causes in a new build output cold.
→
📖
Study
Analyze examples: extract the diagnostic procedure, learn to collapse duplicates and handle chains.
→
✅
Post-Test
Answer with own study notes + original worked examples.

Difficulty Dimensions

DimensionLevelsWhat Varies
Complexity independent, chained, redundant Independent: errors don’t affect each other. Chained: fixing one would expose another (but the count is still N distinct root causes). Redundant: multiple symptoms of the same root cause—the model must not over-count.
Evidence few (3), mid (4), many (5) Number of worked examples demonstrating the diagnostic procedure. More examples = more evidence to learn from.
Seeds 0, 1, 2 Independent random generations for statistical robustness.

Total: 3 complexity × 3 evidence × 3 seeds = 27 items

Dataset Generation

Error lines are generated from template pools modelling real TypeScript compiler errors (TS2345 type mismatches, TS2304 missing imports, TS18047 null accesses, TS2741 missing properties, TS6133 unused variables, TS2769 overload failures) and Jest test failures. Each template is paired with a root-cause key so the generator can control whether multiple lines point to the same underlying issue (redundant) or different ones (independent). For chained conditions, errors are ordered so that earlier fixes would expose later symptoms. Worked examples show a build output, the full enumeration of distinct root causes, and the count.

Examples

Independent Errors

The model sees worked examples of multi-error build outputs with full enumerations, then must count root causes in a new output:

src/auth/session.ts:42:8 - error TS2345: Argument of type ‘string’ is not assignable to parameter of type ‘UserProfile’.
src/api/orders.ts:91:12 - error TS2304: Cannot find name ‘config’. Did you forget to import from ‘@/lib/config’?
FAIL test/authflow.test.ts > AuthFlow > returns 200 on success
Question: How many distinct root causes are there?
Expected: 3

Each error line is independent—a type mismatch, a missing import, and a test failure. The model must enumerate all three, not stop at the first.

Redundant Errors (same root cause, multiple symptoms)

Multiple error lines trace back to fewer distinct root causes:

src/models/user.ts:23:5 - error TS2345: Argument of type ‘string’ is not assignable to parameter of type ‘OrderItem’.
src/pages/dashboard.tsx:67:10 - error TS2769: No overload matches this call. Overload 1 expected ‘OrderItem’ but got ‘string’.
src/api/users.ts:45:3 - error TS18047: ‘formatDate’ is possibly ‘null’.
Question: How many distinct root causes are there?
Expected: 2

The first two errors share the same root cause (type mismatch on OrderItem), while the null access is separate. Models that count lines instead of root causes will over-count.

Scoring

  • Metric: Post-learning accuracy (exact integer match).
  • Extraction: Last integer on its own line, stripped of markdown and punctuation.
  • Assertion: kbench.assertions.assert_equal(post_extracted, str(expected))

What Discriminates Models

  • Independent/few: Most models handle straightforward enumeration of 2–3 unrelated errors—this is a baseline.
  • Redundant: Models must collapse multiple symptoms into shared root causes. Over-counters fail here, revealing whether the model truly learned the diagnostic pattern or just counts error lines.
  • Chained: Errors are causally linked. Models that collapse the entire chain into “just the first error” under-count. The correct answer is still N distinct root causes.
  • Study quality matters: Models that extract a clear, reusable diagnostic checklist from the worked examples outperform those that produce vague summaries.

Prior Work & References

Batch error diagnosis draws on research in automated fault localization, root cause analysis, and multi-fault program repair. No existing benchmark evaluates whether LLMs can learn a diagnostic enumeration procedure from worked examples and transfer it to new multi-error outputs.

Root Cause Analysis & Fault Localization

  • Kang et al. (2024). “Finding the Needle in the Crash Stack: Industrial-Scale Crash Root Cause Localization with AutoCrashFL.” Proposes an LLM agent for crash root cause localization given only the crashdump and source repository—demonstrating the need for systematic enumeration rather than surface-level pattern matching.
  • Chen et al. (2025). “TAMO: Fine-Grained Root Cause Analysis via Tool-Assisted LLM Agent.” Uses multi-modality observation data for fine-grained RCA in cloud-native systems, showing that LLMs benefit from structured diagnostic procedures—the same principle our study phase exploits.

Multi-Fault Program Repair

  • Xiang et al. (2024). “Towards Practical and Useful Automated Program Repair for Debugging.” Found that ~49.5% of multi-location bugs are multi-fault bugs, yet current APR techniques repair at most 8 of 118 single-fault multi-location bugs—highlighting the gap our benchmark targets.
  • Wang et al. (2024). “On the Role of Fault Localization Context for LLM-Based Program Repair.” A large-scale empirical study showing how fault localization context granularity affects LLM repair quality. Our benchmark isolates the upstream problem: can the model even enumerate all faults before attempting repair?

Observational Learning in LLMs

  • Agarwal et al. (2025). “Learning from Observation: A Survey of Recent Advances.” Surveys state-only imitation learning (learning from observed sequences without action labels)—our worked-example paradigm is a text analogue of this framework.
  • Majumdar et al. (2024). “LMAct: A Benchmark for In-Context Imitation Learning with Long Multimodal Demonstrations.” Evaluates in-context imitation learning across zero-, few-, and many-shot regimes. Our evidence dimension (3/4/5 examples) tests a similar axis in a text-only setting.

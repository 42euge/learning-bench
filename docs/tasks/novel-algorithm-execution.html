Novel Algorithm Execution

Procedural Learning

What It Tests

Can the model learn and correctly execute a novel algorithm from a written specification alone? This isolates procedural learning: the ability to internalize a multi-step computational procedure that does not exist in training data and execute it faithfully on new inputs. Nine invented algorithms—from simple conditional swaps to stateful multi-stack operations—ensure the model cannot rely on memorized implementations.

Three-Phase Framework

📝
Pre-Test
Read an algorithm specification and execute it on a test input cold.
→
📖
Study
Rewrite the algorithm step-by-step, identify edge cases, work through a practice example.
→
✅
Post-Test
Execute with own study notes + original specification.

Difficulty Dimensions

DimensionLevelsWhat Varies
Complexity simple, medium, complex Simple: 2-step algorithms (e.g., conditional swap + sum). Medium: multi-pass pipelines with XOR, filters, or chunk rotations. Complex: stateful accumulators with parity gates, threshold resets, and multi-stack collapse rules.
Clarity verbose, standard, terse Verbose: step-by-step with worked examples in the spec. Standard: concise numbered steps. Terse: single-sentence descriptions—the model must unpack compressed instructions.
Seeds 0, 1, 2 Independent random generations for statistical robustness. Each seed selects a different algorithm from its complexity tier.

Total: 3 complexity × 3 clarity × 3 seeds = 27 items

Dataset Generation

Nine novel algorithms are implemented in Python, each operating on a list of integers and returning a single integer. The algorithms are named with invented names (Crinkle Sum, Zigzag Product, Mirror Count, Vortex Reduce, Cascade Filter, Ripple Hash, Nexus Fold, Helix Weave, Prism Stack) to prevent training-data lookup. Each algorithm has three specification variants at different verbosity levels. Test inputs are randomly generated lists of 6–8 integers in the 1–20 range, and ground-truth answers are computed by executing the reference implementation.

Examples

Simple — Crinkle Sum (verbose)

The model receives a full step-by-step specification with a worked example:

Step 1: Go through consecutive pairs left to right. If BOTH are odd, swap them.
Step 2: Sum all elements at even indices (0, 2, 4, …).
Example: [3, 7, 4, 5] → pair(0,1) both odd, swap → [7, 3, 4, 5]; sum arr[0]+arr[2] = 7+4 = 11.
Input: [14, 5, 17, 8, 3, 12]
Expected: A single integer (deterministic based on the algorithm)

Complex — Nexus Fold (terse)

At the hardest difficulty, the spec is a single compressed sentence:

Nexus Fold: acc=0; per element: add if acc even, subtract if odd; reset to x if >20, -x if <-20; final: |acc|.
Input: [10, 7, 3, 15, 4, 6, 18]
Expected: A single integer (the model must unpack the parity gate, overflow resets, and absolute-value transform from one line)

The model must correctly parse that “add if acc even” means checking the accumulator’s parity before each operation, and that overflow resets to the current element, not a fixed value. Terse specs expose whether the model truly understands or just pattern-matches from verbose descriptions.

Scoring

  • Metric: Post-learning accuracy (exact integer match).
  • Extraction: Last integer on its own line, stripped of markdown and punctuation.
  • Assertion: kbench.assertions.assert_equal(post_extracted, str(expected))

What Discriminates Models

  • Simple/verbose: Most models correctly execute 2-step algorithms when given detailed specs with worked examples—this is a baseline.
  • Complex/terse: Requires simultaneously unpacking compressed instructions and maintaining multi-step stateful computation with conditional resets. This is where weaker models break down.
  • Study quality matters: Models that rewrite the algorithm in their own words, identify tricky edge cases (e.g., “the parity check is on the accumulator, not the input”), and verify with a practice example show the largest pre-to-post gains.
  • Arithmetic precision: Complex algorithms like Helix Weave require many intermediate calculations. Models that lose track of running state produce wrong answers even when they understand the algorithm conceptually.

Prior Work & References

Novel algorithm execution bridges program synthesis, code reasoning, and procedural reasoning in LLMs. Existing benchmarks test code generation or understanding—our task tests whether the model can act as an executor of a procedure it has never seen.

Code Reasoning & Execution Benchmarks

  • Gu et al. (2024). “CRUXEval: A Benchmark for Code Reasoning, Understanding and Execution” (ICML 2024). 800 Python functions with input/output prediction tasks. Found that even GPT-4 with chain-of-thought achieves only 75–81% on short functions. Our benchmark goes further by using novel algorithms with no training-data leakage.
  • Zhuo et al. (2025). “CRUXEval-X: A Benchmark for Multilingual Code Reasoning, Understanding and Execution.” Extends CRUXEval to 19 languages. Confirms that code execution reasoning is a fundamental challenge across languages, not just a Python-specific problem.

Procedural Reasoning in LLMs

  • Stechly et al. (2025). “The Illusion of Procedural Reasoning: Measuring Long-Horizon FSM Execution in LLMs.” Used finite-state machine execution as a minimal framework for evaluating procedural reasoning. Found that LLMs can mimic algorithmic behavior for short traces but fail as procedural depth grows—directly relevant to our complexity dimension.
  • Kumar et al. (2025). “Evaluating LLMs’ Reasoning Over Ordered Procedural Steps.” Showed that model performance declines with increasing sequence length and step displacement in procedural tasks—our algorithm complexity and terse specifications probe the same degradation pattern.

Program Synthesis & Algorithm Induction

  • Xu et al. (2025). “CodeARC: Benchmarking Reasoning Capabilities of LLM Agents for Inductive Program Synthesis.” Evaluates agents that interact with hidden target functions by querying inputs and synthesizing candidate programs. Among 18 models, the best achieves only 52.7%—showing that algorithm induction remains hard. Our task flips the direction: the algorithm is given, and the challenge is faithful execution.
  • Shojaee et al. (2025). “Structured Program Synthesis using LLMs: Results and Insights from the IPARC Challenge.” Evaluated structured inductive programming with LLMs on synthetic tasks requiring sequence, selection, and iteration—the same computational primitives our algorithms use.

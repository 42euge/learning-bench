Stale State Recovery

Procedural Learning

What It Tests

Can the model learn—from a small number of worked examples—a world model in which references to state become stale after mutation, and then apply that model to debug and repair a new operation sequence? This isolates procedural learning of state-invalidation rules: the model must internalize which operations invalidate refs, detect the stale-reference bug, and insert a re-snapshot at the correct position. The real-world failure mode is agents that re-use stale handles after page navigations, file mutations, or deployments—producing errors like Ref e803 not found or File has been modified since read.

Three-Phase Framework

📝
Pre-Test
See worked examples of valid op sequences, then fix a buggy sequence cold.
→
📖
Study
Extract staleness rules: which ops invalidate refs, what recovery looks like, which named ops mutate.
→
✅
Post-Test
Fix the buggy sequence with own study notes + original examples.

Implicit: named operations like write_file(), navigate(), commit(), and deploy() secretly invalidate refs as a side-effect—the model must learn which ops mutate from the worked examples.

Total: 3 complexity × 3 evidence × 3 seeds = 27 items

Dataset Generation

Operation sequences are built from a small abstract world model: snapshot() captures state and returns a ref token, act(ref.item_X) uses a ref (fails if stale), mutate() invalidates all prior refs. For the implicit_mutation variant, named operations (write_file, navigate, commit, deploy) secretly mutate, while others (read_file, inspect, log, ping, hash) are pure. Worked examples show valid sequences with OK/FAIL annotations. Test inputs are buggy sequences with one missing re-snapshot; the expected answer is the corrected sequence with a re-snapshot inserted and steps renumbered.

Examples

Single Mutation

The simplest case: one mutation invalidates the initial ref, and the model must insert a re-snapshot:

1. s1 = snapshot()
2. act(s1.item_A)
3. mutate()
4. act(s1.item_B)  ← STALE: s1 was invalidated by mutate()
Corrected sequence:
1. s1 = snapshot()
2. act(s1.item_A)
3. mutate()
4. s2 = snapshot()  # re-snapshot after mutation
5. act(s2.item_B)

Implicit Mutation

Named operations secretly invalidate state. The model must learn from examples which ops mutate:

1. s1 = snapshot()
2. act(s1.item_C)
3. read_file()  ← pure: refs remain valid
4. write_file()  ← MUTATING: invalidates all prior refs
5. act(s1.item_D)  ← STALE: s1 was invalidated by write_file()
Corrected sequence:
1. s1 = snapshot()
2. act(s1.item_C)
3. read_file()
4. write_file()
5. s2 = snapshot()  # re-snapshot after mutation
6. act(s2.item_D)

The model must distinguish read_file() (pure) from write_file() (mutating) based solely on evidence in the worked examples—there is no explicit rule given for which named operations mutate.

Scoring

  • Metric: Post-learning accuracy (exact sequence match after normalization).
  • Extraction: Longest contiguous block of numbered operation lines, normalized (step numbers stripped, comments removed, whitespace collapsed, lowercased).
  • Assertion: Extracted sequence must match the normalized expected sequence exactly.

What Discriminates Models

  • Single mutation: Models that understand basic state invalidation handle this. It serves as a baseline and sanity check.
  • Multiple mutations: Only one of 2–3 mutations is missing its refresh. Models must track ref validity across the entire sequence, not just insert a refresh after every mutation.
  • Implicit mutation: The hardest condition. Models must learn from examples which named operations are secretly mutating—a pure inductive learning challenge. Models that treat all named ops as safe (or all as mutating) fail.
  • Study quality matters: Models that extract an explicit checklist of which operations invalidate refs (citing evidence from the worked examples) show the largest pre-to-post improvement on implicit_mutation items.

Prior Work & References

Stale state recovery draws on the classical frame problem in AI, modern agent state-tracking research, and software engineering benchmarks that test planning with side effects.

The Frame Problem & State Tracking

  • McCarthy & Hayes (1969). “Some Philosophical Problems from the Standpoint of Artificial Intelligence.” Introduced the frame problem: how does an agent determine which aspects of state change and which persist after an action? Our benchmark is a direct instantiation—the model must learn which operations change the world (invalidating refs) and which leave it intact.
  • Li et al. (2025). “StateAct: Enhancing LLM Base Agents via Self-prompting and State-tracking.” Introduces “chain-of-states,” extending chain-of-thought with explicit state tracking over time. Our task evaluates whether models can learn state-tracking rules from examples without needing architectural modifications.

Agent Failures & Stale Context

  • Chen et al. (2025). “Your LLM Agents are Temporally Blind: The Misalignment Between Tool Use Decisions and Human Time Perception.” Found that LLM agents over-rely on stale context and skip needed tool calls when time elapses—the temporal analogue of our stale-ref problem. Our benchmark isolates the structural (not temporal) version of this failure.
  • Agarwal et al. (2025). “RefactorBench: Evaluating Stateful Reasoning in Language Agents Through Code.” Identified failure modes in tracking past actions, showing 43.9% improvement when agents condition on state representations. Our task measures whether the state-tracking rules themselves can be learned from examples.
  • Zhang et al. (2025). “Beyond Completion: Probing Cumulative State Tracking to Predict LLM Agent Performance.” Showed that cumulative state-tracking ability is predictive of overall agent performance, directly motivating benchmarks that isolate state-tracking as a learnable skill.

Software Engineering Benchmarks

  • Jimenez et al. (2024). “SWE-bench: Can Language Models Resolve Real-world Github Issues?” (ICLR 2024). The leading benchmark for LLM software engineering. SWE-bench tasks implicitly require state management (understanding what changed in a codebase), but do not isolate or measure state-tracking as a learnable skill. Our task complements SWE-bench by providing a controlled, minimal evaluation of this specific capability.
  • Yang et al. (2025). “SWE-Bench Pro: Can AI Agents Solve Long-Horizon Software Engineering Tasks?” Extends SWE-bench to long-horizon tasks where state management across many steps becomes critical. Our benchmark provides a focused diagnostic for the state-tracking sub-skill that these longer tasks require.

Planning with Side Effects

  • Wang et al. (2025). “Where LLM Agents Fail and How They Can Learn From Failures.” Catalogues common LLM agent failure modes, including failure to account for side effects of previous actions. Our benchmark turns this failure mode into a measurable, learnable task.

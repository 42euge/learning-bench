# Trace-Based Imitation

**Sub-ability:** Observational Learning
**Difficulty grid:** 3 complexity × 3 trace counts × 3 seeds = 27 items

## What it tests

Can the model learn a novel procedure by observing worked examples (execution traces), then apply it to a new input? This tests observational learning — inferring a procedure from demonstrations without explicit instruction.

## Three-phase framework

1. **Pre-test:** The model sees worked examples of a procedure applied to number lists (showing input, intermediate steps, and output), and must apply the same procedure to a new input. No analysis time.
2. **Study:** The model analyzes the traces to write down the exact procedure — specifying order of operations, conditions, and how each step transforms the data.
3. **Post-test:** The model receives its own procedure notes plus the original traces, and must apply the procedure to the same test input.

## Difficulty dimensions

| Dimension | Levels | What varies |
|-----------|--------|-------------|
| **Complexity** | simple, medium, complex | Simple: 1-step procedures (zigzag_sum, mirror_diff, triangle_count). Medium: 2-step compositions (spiral_crunch, wave_hash, cascade_mod). Complex: 3+ steps with conditional branching (branch_fold, pivot_weave, chain_gate). |
| **Traces** | 2, 3, 4 | Number of worked examples provided. More traces = easier to identify the procedure. |
| **Seeds** | 0, 1, 2 | Independent random generations. |

## Dataset generation

Nine novel procedures are defined with increasing complexity. Each procedure transforms a list of integers into a single integer through specific operations. Worked traces are generated by running each procedure on random inputs and formatting each step explicitly. The test input is always different from any traced input.

**Example procedures:**
- `zigzag_sum`: Alternate adding and subtracting elements left-to-right
- `wave_hash`: Sort, take differences of consecutive elements, then sum
- `branch_fold`: If first element is even, fold left with addition; else fold right with subtraction

## Scoring

- **Metric:** Post-learning accuracy (exact integer match).
- **Extraction:** Last integer found in the response (handles negative numbers).
- **Per-item assertion:** `kbench.assertions.assert_equal(post_extracted, str(expected))`

## What discriminates models

- **Simple/4 traces:** Most models can identify single-step operations from 4 examples.
- **Complex/2 traces:** Conditional branching procedures with only 2 traces require genuine procedural reasoning. The model must figure out what triggers different code paths.
- **Precision matters:** Procedures like `cascade_mod` involve modular arithmetic chains where off-by-one errors cascade. Models that approximately understand the procedure still get the wrong number.

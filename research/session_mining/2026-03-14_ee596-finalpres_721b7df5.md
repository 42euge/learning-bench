## Key Failure Types

### 1. False Positive Error Detection (NOVEL)
The `is_error` heuristic (checking if result contains "error"/"failed"/"rejected") produces massive false positive rates when the agent reads files that discuss error handling. 14/15 "errors" in this session are FPs. This is a **metrology problem** -- the measurement instrument is unreliable for sessions involving error-related content.

### 2. User-Driven Thrashing vs Agent-Driven Thrashing
The high thrashing score (0.65) is misleading. It reflects a human iterating on visual tweaks, not an agent stuck in a failure loop. The agent succeeds on every attempt. Current thrashing metrics cannot distinguish between:
- (a) Agent repeatedly failing and retrying the same resource
- (b) User repeatedly requesting changes to the same resource

### 3. Write-vs-Edit Strategy Selection
Agent defaults to `Write` (full overwrite) for large changes and `Edit` for small ones. But 6 full overwrites of a ~10K char file when only specific sections changed means wasted tokens and lost change traceability. This is not a failure per se, but a suboptimal tool selection pattern.

### 4. Ad-hoc Notebook Cell Extraction
Rather than reading the entire notebook once (or using a structured extraction), the agent writes bespoke Python scripts 11 times to pull individual cells. Each script is slightly different (different cell indices, different fields). A "read notebook structure once, query in memory" approach would be more efficient.

## Potential Benchmark Abstractions

1. **Error detection calibration** -- Metacognition: given tool outputs that mention "error" in their content, distinguish between actual tool failures and content that discusses errors. Tests whether a model can separate meta-level signals from object-level content.

2. **Thrashing source attribution** -- Metacognition: given a sequence of repeated resource accesses, classify whether repetition is driven by (a) agent failure recovery, (b) user-directed iteration, or (c) unnecessary redundancy. Requires understanding intent behind action sequences.

3. **Granularity selection** -- Skill selection: given a file and a desired change, choose between full rewrite (Write) and targeted edit (Edit). Tests whether the model matches tool granularity to change scope.

4. **Batch vs serial data extraction** -- Procedural learning: given a structured data source (notebook, spreadsheet) that will be queried multiple times, learn to extract data in bulk rather than via repeated one-off scripts. Tests planning horizon and resource efficiency.

5. **Cosmetic convergence under vague feedback** -- Concept formation: user gives ambiguous aesthetic feedback ("fix this", "change to something else", "idk something else lol"). Agent must converge on acceptable output despite underspecified criteria. Tests handling of ill-defined objectives.
